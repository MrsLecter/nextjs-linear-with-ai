### Ограничения:
- не было достаточно времени все протестировать, поэтому возможны баги
- нет тестов
- локальная бд

### Seed db

В скриптах есть скрипт для заполнения бд. По каждой задаче свой объем данных - это позволит создать более точное впечатление о работе функций и даст возможность лучше протестировать. Идея такая, запускаем seed для нужной задачи, тестируем, очищаем бд, снова заполняем уже под другую задачу.


### Алгоритм приоритизации таски

Тут смысл чтоб сформировать начальные правила которые позволят отсеить большинство нерелевантных задач. Тут набор полей очень ограничен, поэтому механизм составление первоначальной оценки размыт. Больший приоритет отдается ИИ саммаризации title + description. Соответственно более умная модель даст лучший результат. Тут для соблюдения баланса стоимость-скорость выбран gpt-5.4-mini

Алгоритм сначала отбирает только незавершённые задачи, затем для каждой считает базовый приоритетный балл на основе трёх сигналов: уровня priority, текущего status и возраста задачи по createdAt; после этого задачи ранжируются по этому baseline score, а LLM поверх ранжирования делает только смысловую дооценку — анализирует title и description, чтобы понять, есть ли признаки срочности, блокера, бага, высокой продуктовой важности или, наоборот, расплывчатости и низкой actionable-ценности; в итоге выбирается одна задача, которая выглядит наиболее важной и уместной именно сейчас, при этом завершённые задачи никогда не рассматриваются, а если AI-оценка недоступна или невалидна, система детерминированно берёт задачу с максимальным baseline score.

Тут может случится такой кейс что есть несколько важных задач. Одна задача неявно блокирует другую, например если сломаны миграции мы не можем исправить баг с логином пользователя. Но этот кейс больше ясен разработчикам чем ИИ. Поэтому отображается несколько срочных тасок. Также могут быть связанные задачи по тому же принципу.

Ну и то же, если в описании писать URGENT, BLOCKER, CRITICAL, ASAP, revenue impact И т.д. агент начнёт это переоценивать. Поэтому модель должна оценивать не тон текста, а обоснованность(влияние, что сломано, бизнес эффект, дедлайн)

# Алгоритм декомпозиции

Тут важно свормировать четкие критерии по которым можем разбить задачу, по которым задача непонятна(current behaviour, expected behavior, не написанно что ломается и какой scope).

Насчет используемых функций, тут используется tool calling.

Алгоритм декомпозиции работает так: пользователь в Edit Task modal редактирует title и description и нажимает Generate subtasks, после чего фронт отправляет на backend текущий draft формы, а не старые данные из БД; backend запускает AI agent через OpenAI Responses API с function calling, где первым обязательным шагом модель вызывает assess_task_decomposition и классифицирует задачу как needs_clarification, если не хватает контекста, cannot_decompose, если задача уже маленькая и атомарная, или ready, если она понятна и содержит несколько осмысленных инженерных шагов; только в случае ready агент вызывает generate_subtasks, получает структурированный список подзадач, backend валидирует все входы и AI-результаты через zod и возвращает на фронт только preview без записи в БД, а сохранение происходит отдельным действием — только если пользователь явно нажимает Create subtasks.
